The newest models have made it possible to accelerate various phases of the traditional SDLC (software development lifecycle). At a high-level, SDLC involves ideating, creating a PRD (product requirement doc), UX mockups, a technical brief and implementing and testing the code. AI-based SDLC frameworks such as BMAD and Speckit have implemented each stage as AI skills. The output of each stage are passed along the SDLC pipeline. Recently, Google also published a [short document](https://www.kaggle.com/whitepaper-the-new-SDLC-with-vibe-coding) describing this new paradigm.

Unlike Anthropic where every employee is a Member of technical staff, many firms still have defined roles. In these cases, different people own each stage and pass the outputs along. For example, PMs still own the initial stages while engineers own the implementation stages. Close collaboration and alignment, quality checks and signoffs are important for most brownfield projects. These checks remain inherently human in nature so the ideation, PRD, UX and tech briefs still feel like traditional SDLC. However, once the tech brief is broken down into a list of tasks implementation can be sped up.

A list of tasks can be implemented rapidly by AI. Earlier this year, Ralph loop was a popular technique to implement this checklist. An external (bash) `while` loop would pickup the tasks, implement one, erase the agent's context and pickup the next task. The context was erased because as Claude would implement 1 task, all the intermediate data would have be fed-back to the LLMs because LLMs are completely stateless. If you don't erase it's memory, it would get confused over time and create gargage output.

Recently this "loop" idea has been re-imagined natively inside the harnesses (claude code or codex) via the `goal` command. In Claude, after a task is implemented, the agent asks a **separate evaulator model** to run a quality check (humans need to define this). If the checks fail, the agent continues to iterate against the **implementation model** to continue fixing all the issues until the evaluation passes (assuming auto-mode is turned on). This evaluation step is a clever improvement as opposed to assuming that the implementation model completed it's task perfectly.

Internal employees of OpenAI call the `goal` command "Ralph 2.0" but this is inaccurate. The key difference is that `goal` runs within 1 session while the Ralph loop resets sessions, which erases context. This is actually important for a really long task list because Ralph is basically using the task file to maintain state while `goal` is maintain state within the context, which runs into the context rot issue I described earlier. `goal` shines if the tasklist is shorter or if the evaluation isn't machine-verifiable. Machine verifiable refers to deterministic verification like passing linting and tests. But if the evaluation needs human judgement `goal` can be a better option.
