The newest models have made it possible to accelerate various phases of the traditional SDLC (software development lifecycle). At a high-level, SDLC involves ideating, creating a PRD (product requirement doc), UX mockups, a technical brief and implementing and testing the code. AI-based SDLC frameworks such as [BMAD](https://github.com/bmad-code-org/bmad-method) and [Speckit](https://github.com/github/spec-kit) have re-imagined each stage as AI skills that can be triggered by users. The output of each stage are passed along the SDLC pipeline. Recently, Google also published a [short document](https://www.kaggle.com/whitepaper-the-new-SDLC-with-vibe-coding) describing this new paradigm.

![_config.yml]({{ site.baseurl }}/images/sdlc.png)

Unlike Anthropic where every employee is a Member of technical staff, many firms still have defined roles. In these cases, different people own each stage and pass the outputs along. For example, PMs still own the initial stages while engineers own the implementation stages. Close collaboration and alignment, quality checks and signoffs are important for most brownfield projects. These checks remain inherently human in nature so the ideation, PRD, UX and tech briefs still feel like traditional SDLC. However, once the tech brief is broken down into a list of tasks implementation can be sped up.

e.g., of a spec.md for a hotel trip planner app which is a PRD-style artifact. This is converted to a tech brief and then broken down into task.md.

A list of tasks can be implemented rapidly by AI. Earlier this year, Ralph loop was a popular technique to implement this checklist. An external (bash) `while` loop would pickup the tasks, implement one, erase the agent's context and pickup the next task. The context was erased because as Claude would implement 1 task, all the intermediate data would have be fed-back to the LLMs because LLMs are completely stateless. If you don't erase it's memory, it would get confused over time and create gargage output, a problem called context rot.

Recently this "loop" idea has been re-imagined natively inside the harnesses (claude code or codex) via the `goal` command. In Claude, after a task is implemented, the agent asks a **separate evaulator model** to run a quality check (humans need to define this). If the checks fail, the agent continues to iterate against the **implementation model** to continue fixing all the issues until the evaluation passes. This evaluation step is a helpful convenience for end-users to define quality gates. 

![_config.yml]({{ site.baseurl }}/images/ralph.png)

Internal employees of OpenAI call the `goal` command their own take of the Ralph loop but this misses the point because `goal` runs within 1 session while the Ralph loop resets sessions to erase context. The context erasure is important for long task lists to avoid context rot. `goal` shines if the tasklist is shorter or if the evaluation isn't machine-verifiable (see image, courtesy of [Chase AI](https://www.youtube.com/watch?v=JirDfgJcJFU)). For machine verifiable tasks (tiers 1-3), Ralph loop is simple, long running and effective. But for tasks that need judgement (tiers 3-5) non-deterministic instructions can be encoded as instructions into a `goal` command.

![_config.yml]({{ site.baseurl }}/images/ai-verification.png)

> NOTE: `goal` is not to be confused with the `loop` command, which is just a scheduled task that runs periodically and is scoped to a Claude session (only runs when the session is alive). Similarly, `goal` is different from `schedule` (routines) which run remotely on Anthropic's cloud environment, executing agentic instructions based on various triggers.
