The newest models have made it possible to accelerate various phases of the traditional SDLC (software development lifecycle). At a high-level, SDLC involves ideating, creating a PRD (product requirement doc), UX mockups, a technical brief and implementing and testing the code. AI-based SDLC frameworks such as BMAD and Speckit have implemented each stage as AI skills. The output of each stage are passed along the SDLC pipeline. Recently, Google also published a [short document](https://www.kaggle.com/whitepaper-the-new-SDLC-with-vibe-coding) describing this new paradigm.

Unlike Anthropic where every employee is a Member of technical staff, many firms still have defined roles. In these cases, different people own each stage and pass the outputs along. For example, PMs still own the initial stages while engineers own the implementation stages. Close collaboration and alignment, quality checks and signoffs are important for most brownfield projects. Because AI hasn't automated judgement yet, alignment and quality checks remain inherently human in nature. Hence, the ideation, PRD, UX and tech briefs still feel like a traditional SDLC. However, once the tech brief is broken down into a list of tasks implementation can be sped up (note - typically, it's impossible to know all the tasks up-front and new complexities are usually discovered as projects get implemented).

A list of tasks can be implemented rapidly by AI. Earlier this year, Ralph loop was a popular technique to implement this checklist. An external (bash) `while` loop would pickup the tasks, implement one, erase the agent's context and pickup the next task. The context was erased because as Claude would implement 1 task, all the intermediate data would have be fed-back to the LLMs because LLMs are completely stateless. If you don't erase it's memory, it would get confused and create gargage output.

![_config.yml]({{ site.baseurl }}/images/ralph.png)

Recently this "loop" idea has been implemented natively inside the harnesses (claude code or codex) via the `goal` command. In Claude, after a task is implemented, the agent asks a **separate** evaulator model whether a quality check was passed (humans need to define this). If not, the agent asks the main implementation model to continue fixing all the issues until the evaluation was passed. This evaulation step is a clever improvement comparing to assuming that the main agent implemented the task correctly. Most of the time, the main agent makes a lot of mistakes and needs to be reined in to improve quality. Does the `goal` command invalidate the Ralph loop? In some ways, they're different ideas - the goal agent is optimizing 1 task while Ralph loop works over `n` tasks. But because harnesses have become so much better at managing context and LLMs are also more capable, the `goal` command run internally inside the harness can replace the externally run Ralph loop.
