[Code execution with MCP](https://www.anthropic.com/engineering/code-execution-with-mcp) is an informative article about managing the size of agentic context windows. When context windows become too large, the quality of output decreases because the agent stores too much irrelevant information. Two issues are described. The first issue is that if you have x MCP definitions, then all of them have to be loaded into the agent context which doesn't scale. The second issue is that when an agent makes multiple MCP calls (e.g., load doc from google drive and upload it to salesforce), then the intermediate results have to be sent back to it (e.g., the doc is the intermediate result). When these results are huge (e.g., the doc is large) then the context window is used up, and the agent can't return an output. For example, if you ask claude to "design youtube", the agent will likely error out.

The solution proposed for both problems is to enable the agent to write code. For example, in vscode, or even in online webapps like Gemini or ChatGPT, agents can write code and execute the logic directly on the IDE or web UI. Since agents can now write code, we can expose a /search API endpoint so the agents can discover the list of MCP tools. This avoids loading all MCP tool definitions into the context window. Also, a key point is that agents can write code to filter out irrelevant intermediate output. This further reduces the context window.

When agents can code, a highly useful pattern emerges where we allow an agent to **persist state in files**. For example, if the agent has 5 steps, it can save the state after each step in a file. This makes the agent highly scalable because the state can be paused and resumed. Importantly, the agent doesn't need to maintain a large context over those 5 steps as it can save it's intermediate state outside it's context window. Those individual steps can also be abstracted and saved into files as skills (e.g., brainstorming skill) and you can ask the main agent to use those skills in future.

Newer approaches use subagents to manage context windows. Subagents are processes that can do independent work and report back to the main claude agent. You as a user can create subagents that have specific roles and personas like a technical design architect or product manager. Thereafter, you can ask the main agent questions and it can choose to use one or many subagents to solve a task. These subagents have their own context window so they don't need to store all the useless context that the main agent accumulates over time. They're great for parallel work because the main agent can call multiple subagents to do tasks. For example, you can create a technical architect subagent that can "design the youtube API" and another tester subagent that can "write tests for this API". Then if you ask the main agent to "design youtube", it will hopefully coordinate these 2 subagents to return an output. Subagents aren't just for developers - for example Gemini gems are available for consumers to use.

There are also ways to formally store the coordination logic and call it explicitly with a command e.g., /techbrief command can be created that explicitly coordinates specific agents. Now you can explicitly ask claude "/techbrief design youtube" and the main agent will load the orchestration instructions and follow the defined workflow. This way, you can force the main agent to orchestrate the two subagents in a deterministic way rather than rely on hope.

Github libraries such as "BMAD" or "Get shit done" are collections of commands, subagents and skills that can be supplied to the main agent. I have used commands to perform market research, review tech designs, review PRs and write PRs. I found that commands are effective as long as the incremental output is reviewed by me step-by-step. My main gripe is that the best reasoning models take a few minutes to return useful output, which feels disruptive. I really hope that thinking models get faster over time.
