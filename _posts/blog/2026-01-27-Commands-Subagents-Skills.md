[Code execution with MCP](https://www.anthropic.com/engineering/code-execution-with-mcp) is a fantastic article about scale. It explains that MCP allows agents to make tool calls, but there are serious problems with scale. The first issue is that if you have x MCP definitions, then all of them have to be loaded into the LLM, which occupies space in the LLM context window. Remember that each LLM has a limited context window (e.g., 10,000 tokens but depends on model), and if you ask it too many questions, it will hit the max and error out. 

The second issue is that when an agent makes multiple MCP calls (e.g., load doc from google drive and upload it to salesforce), then the intermediate results have to be sent back to the LLM (e.g., the doc is the intermediate result). When these results are huge (e.g., the doc is large) then the context window is used up, and again the LLM can't return an output. I tried this myself - I asked claude to "design youtube". The main agent ran out of it's context window and didn't return any result.

The solution proposed for both problems is to enable the agent to write code. For example, in vscode, or even in online webapps like Gemini or ChatGPT, agents can write code and execute the logic directly on the IDE or web UI. Since agents can now write code, we can expose a /search API endpoint so the agents can discover the list of MCP tools. This avoids loading all MCP tool definitions into the context window. Also, a key point is that **agents can write code to filter out irrelevant intermediate output**. This further reduces the context window which is highly useful because if the context window has too much data, the agents return low quality output because most of the data is usually irrelevant to the task at hand.

When agents can code, a highly useful pattern emerges where we allow an agent to persist state in files. For example, if the agent has 5 steps, it can save the state after each step in a file. This makes the agent highly scalable because the state can be paused and resumed. The agent doesn't need to maintain a huge context over those 5 steps, and it can actually help me incrementally "design youtube". Those individual steps can also be abstracted and saved into files as skills (e.g., brainstorming skill) and you can ask the main agent to use those skills in future.

So far, the article has discussed this improvements from the perspective of the main agent, which is what most people interact with when they use ChatGPT or the chat function in their IDE. Anthropic applied similar ideas an developed enchancements to the Claude main agent such as subagents and commands.

Subagents are processes that can do indepdendent work and report back to the main claude agent. You as a user can create subagents that have specific roles and personas like a technical design architect or product manager. Thereafter, you can ask the main agent to trigger and call multiple subagents to solve a task. These subagents have their own context so they don't need to store all the useless context that the main agent accumulates over time. They're great for parallel work because the main agent can call multiple subagents to do tasks. For example, I can create a technical architect subagent that can "design the youtube API" and another tester subagent that can "write tests for this API". Then I can ask the main agent to "design youtube" and it will coordinate these 2 subagents to return an output.

Furthermore, the coordination logic for the technical architect subagent and tester subagent can be written into a file and executed by a "slash command" e.g., /techbrief command which is an instruction template to create tech designs for any user input. Now I can explicitly ask claude "/techbrief design youtube" and the main agent will load my orchestration instructions and follow the defined workflow. This improves my quality of life because I can force the main agent to orchestrate my two subagents in a very specific way. I can also provide custom skills to each subagent.

There are libraries such as "BMAD" and "Get shit done" that can be supplied to the main agent. They enhance the main agent with subagents, commands and skills. I hope to share more insights over time as I experiment with these libraries.
