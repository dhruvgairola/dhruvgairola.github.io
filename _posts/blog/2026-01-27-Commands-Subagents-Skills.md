[Code execution with MCP](https://www.anthropic.com/engineering/code-execution-with-mcp) is a fantastic article about scale. It explains that MCP allows agents to make tool calls, but there are serious problems with scale. The first issue is that if you have x MCP definitions, then all of them have to be loaded into the LLM, which occupies space in the LLM context window. Remember that each LLM has a limited context window (e.g., 10,000 tokens only), and if you ask it too many questions, it will hit the max and error out. 

The second issue is that when an agent makes multiple MCP calls (e.g., load doc from google drive and upload it to salesforce), then the intermediate results have to be sent back to the LLM (e.g., the doc is the intermediate result). When these results are huge (e.g., the doc is large) then the context window is used up, and again the LLM can't return an output. I tried this myself - I asked claude to "design youtube". The main agent ran out of it's context window and didn't return any result.

![_config.yml]({{ site.baseurl }}/images/claude_err.png)

The solution proposed for both is to allow the agent to write code and execute the code in an execution environment. For example, in vscode, or even in online webapps like Gemini, agents write code and execute the logic directly on the IDE or web UI. With this approach, an API endpoint (e.g., /search) can be used to tell the agents which MCP tools exist, which avoids loading all tools into the context window. The agent can then write code to invoke the tools. This solves the first problem described. 

Also, any **intermediate output in the code logic doesn't need to be sent back to the main agent**. Only the final output is returned and all the intermediate output is discarded. This is critical because the main agent can be overwhelmed with too much context - this will cause it to either terminate prematurely or return a low quality result because it has too much irrelevant context.

When agents can code, a highly useful pattern emerges where we allow an agent to persist state in files. For example, if the agent has 5 steps, it can save the state after each step in a file. This makes the agent highly scalable because the state can be paused and resumed. You can also iterate with the agent. For example, I can create a technical architect agent that can "design youtube". I can define an agent that gather requirements, creates an API, a data model, test cases, and ask the agent to save state in a file in each step. As a user, I can additionally create a command that explicitly uses this "architect" agent to do work for me. I can also choose to update my commands file to orchestrate multiple agents.

Recently, I've tried to play with "BMAD" and "Get shit done" repos on github that have implemented ideas from this article. These repos can be installed as plugins and they enhance claude with commands, subagents and skills. We spoke about commands earlier. However, something that I learned was that within command definition files, you can trigger subagents. These subagents are processes that can do indepdendent work and report back to the main orchestrator. You as a user can create subagents that are good at specific roles like technical design or product design. They have specific personas. They're great for parallel work and each subagent has their own context that is discarded after their task is over. This ensures that the main claude process does'nt have to maintain a ton of context because storing a lot of irrelevant context lowers quality (context rot). Finally, the main claude agent can also be equipped with skills (e.g., brainstorming skill) for you as a user to interact with. I hope to share more insights over time.
