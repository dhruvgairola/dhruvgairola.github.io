[Code execution with MCP](https://www.anthropic.com/engineering/code-execution-with-mcp) is a fantastic article about scale. It explains that MCP allows agents to make tool calls, but there are serious problems with scale. The first issue is that if you have x MCP definitions, then all of them have to be loaded into the LLM, which occupies space in the LLM context window. Remember that each LLM has a limited context window (e.g., 10,000 tokens only), and if you ask it too many questions, it will hit the max and error out. 

The second issue is that when an agent makes multiple MCP calls (e.g., load doc from google drive and upload it to salesforce), then the intermediate results have to be sent back to the LLM (e.g., the doc is the intermediate result). When these results are huge (e.g., the doc is large) then the context window is used up, and again the LLM can't return an output. I tried this myself - I created a claude subagent with reasoning mode and asked it to "design youtube". It ran out of then context window and didn't return any result.

![_config.yml]({{ site.baseurl }}/images/claude_err.png)

The solution is to allow the agent to write code and provide an execution environment. For example, in vscode, or even in online webapps like Gemini, it's normal to allow agents to write code and execute the logic directly on the IDE or webapp. With this approach, and API endpoint (e.g., /search) can be used to tell the agents which MCP tools exist, which avoids loading all tools into the context window. The agent can then write code and the **intermediate states are not sent to the LLM context window but instead stay in the execution environment**. This is the critical point. The execution environment has no limitations like "context windows" - you can give it as much memory / CPU as your laptop or server allows. 

A highly useful benefit of this coding approach is that the agent can persist state in files. For example, if the agent has 5 steps, it can save the state after each step in a file. This makes the agent highly scalable because the state is in the execution environment and completely outside of the context window. I can now create an agent that can "design youtube" by creating multiple steps (e.g., gather requirements, design API, data model, testing, etc), and force the agent to save state in a file. Unlike before this agent does not error out and actually produces a useful output when I want it to "design youtube". As a user, I can also crystallize these series of steps into a command file so that I can explicitly tell claude to execute a command to design a technical brief e.g., /designtechbrief.

Recently, I've tried to play with "BMAD" and "Get shit done" repos on github that have implemented ideas from this article. These repos can be installed as plugins and they enhance claude with commands, subagents and skills. We spoke about commands earlier. However, something that I learned was that within command definition files, you can trigger subagents. These subagents are processes that can do indepdendent work and report back to the main orchestrator. You as a user can create subagents that are good at specific roles like technical design or product design. They have specific personas. They're great for parallel work and each subagent has their own context that is discarded after their task is over. This ensures that the main claude process doens't have to maintain a ton of context because storing a lot of irrelevant context lowers quality (context rot). Finally, the main claude agent can also be equipped with skills (e.g., brainstorming skill) for you as a user to interact with. These are fun new concepts and I hope to share more insights over time.
